## 概览

故障的类型：

- 事务故障：逻辑错误，或程序abort。
- 系统崩溃：如断电。通常认为不会破坏磁盘上的非易失性存储内容。
- 磁盘故障：磁盘上数据丢失，通常能检测出来。

恢复算法涉及两部分：

- 事务正常运行时做准备，如日志。
- 故障发生后，根据这些记录把数据库恢复到正确状态。

## 日志

日志记录了数据库修改过程，目的是系统崩溃后还能根据日志判断哪些事务要保留，哪些事务要撤销。一般保存在stable storage（磁盘）上。

一个事务T的日志包括三种：

1. 事务开始时写：`<T start>`
2. 事务修改数据X之前写：`<T X v_old v_new>`
3. 事务成功结束时写：`<T commit>`

## 恢复操作

恢复时有两种可能操作：

- **undo**：使用旧值，撤销未完成事务的修改。撤销修改数据时，写log`<T X v_old>`；结束undo时，写log`<T abort>`。
- **redo**：使用新值，重新执行已经提交事务的修改。不用写log。

Undo和redo都是幂等的，即做一次和做多次效果一样。

日志恢复有两种主要方式：

1. **延迟修改**：事务执行过程中只写日志，不立刻改数据库；等事务提交之后，再根据日志真正更新数据库。崩溃时，如果未提交，则不需要undo；如果已提交，则需要redo。简单，但灵活性差。
2. **立即修改**：事务提交前就将修改写到缓冲区；缓冲区到磁盘的写可发生在任何时刻，写这里的output和缓冲区写的顺序可不同。崩溃时，如果日志有start但没有commit，则说明未完成，需要undo；如果既有start又有commit，则说明已经提交，需要redo。

### 检查点

Checkpoint避免每次从头扫描日志，加快恢复。

每个checkpoint中完成三件事：

1. 把内存里的日志记录强制写到稳定存储。
2. 把已经修改过的缓冲区块写回磁盘。
3. 在日志中写`checkpoint L`，其中L是checkpoint时仍然活跃的事务列表（即已经开始但还没有commit或abort）。

恢复时，系统从日志末尾往前找最近的`checkpoint L`。找到后，不需要考虑更早的事务，只需要处理L中和checkpoint以后的事务即可。

**两阶段恢复**：

1. Redo phase：redo所有事务（即使没提交也重做）。具体方法为，找到最近的checkpoint，将L中事务加入undo list。向后扫描，重做所有的修改操作。如果遇到`Ti start`，将其加入undo list；如果遇到`Tj commit`或`Tj abort`，将其从undo list中删除。
2. Undo phase：undo所有未完成的事务。具体方法为，从后往前扫描，对于undo list中的事务的log，如果遇到修改数据，则改回旧值；如果遇到start，则写入abort日志，并将其从undo list中删除。

Fuzzy checkpointing是一种模糊检查点机制，不要求系统暂停，也不要求一次性把所有脏页都立刻写回磁盘，因此checkpoint是一个模糊的时间区间。主要用于ARIES中。流程为：

1. 暂停所有事务的数据更新，时间很短。
2. 写checkpoint日志，并强制写到磁盘。
3. 检查并列出所有脏数据，不写回磁盘，允许事务执行。
4. 后台慢慢把脏块写回磁盘。
5. 在磁盘上的某个固定位置（last_checkpoint），保存最近一次checkpoint日志记录的位置。

Fuzzy checkpoint恢复时，直接从last_checkpoint开始扫描。

## 并发事务的恢复

真实数据库中，多个事务并发执行。所有事务共享同一个磁盘缓冲区和同一个日志文件，一个缓冲块里也可能包含多个事务修改过的数据。

下面都假设使用strict two-phase locking（事务修改数据时要拿排他锁，而且这个排他锁通常要持有到事务结束，避免级联回滚）。记录checkpoint时更新操作暂停。

**三阶段恢复**：

1. 将redo list和undo list初始化为空。从日志末尾向前扫描到最近的checkpoint，扫描过程中，维护：
   - 如果遇到`T commit`，将其加入redo list
   - 如果遇到`T start`，但T不在redo list中，则加入undo list（如果已提交，则一定先加入redo list）
   - 如果遇到`T abort`，将其加入undo list
   - 对checkpoint的L中的事务T，如果T不再在redo list中，则加入undo list（说明到崩溃时仍未结束）
2. 仍从末尾向前扫描，直到undo list中所有事务最早的start。扫描过程中，撤销undo list中每个事务的操作。
3. 找到最后的checkpoint，从checkpoint向后扫描至末尾。扫描过程中，重做redo list中每个事务的操作。

## 日志缓冲

磁盘写入单位是block，但一条日志记录通常很小，因此先缓存在内存中，等日志块满了或需要强制写日志（如checkpoint）时，再一起写到stable storage。

日志缓冲需要满足：

- log records按顺序写入磁盘，且commit日志最后写。
- Write-Ahead Logging（WAL，先写日志规则）：必须先将日志写到稳定存储，再执行对应的操作，如将数据块写回磁盘、进入commit状态等。

几种策略：

- **no-force policy**：事务提交时，不一定要把它修改过的所有block写回磁盘。该策略下commit快，但如果崩溃时提交事务的数据还没落盘，则需要redo。
- **force policy**：事务提交时，修改的block必须都写回磁盘。
- **steal policy**：缓冲区不够用时，可以把包含未提交事务修改的数据块提前写回磁盘。该策略下buffer管理更灵活，但如果之后事务回滚或系统崩溃，则需要undo。

**latch**为output时短时间持有的内部锁，用来保护内存中的数据块。当buffer block写回磁盘时，需要先拿到这个块的exclusive latch，确保没有事务正在修改它；然后flush相关日志；再把数据块output到磁盘；最后释放latch。

日志缓冲可以在reserved main memory，也可以在虚拟内存中，各自的缺点为：

- reserved main memory：主存中预留出的区域，和数据库buffer等分离，灵活性受限。
- virtual memory（主要实现方式）：DBMS只负责将buffer page和磁盘交流，但操作系统可能写入磁盘中的swap page，此时需要额外从磁盘中读入buffer page，即dual paging problem。解决方法为，OS替换页时不写到自己的swap page，而是写到数据库的页中。


## 非易失性存储损坏

如果发生Failure with Loss of Nonvolatile Storage（非易失性存储损坏），如磁盘损坏，则可能日志和数据库都丢失。解决方法为，定期把整个数据库复制到更可靠的stable storage，这一操作称为**dump**。

Dump流程：

1. 把内存中的日志写到稳定存储，把buffer中的数据块写回磁盘。
2. 将整个数据库复制到稳定存储。
3. 在日志中写入`dump`。

恢复流程：

1. 从最近一次dump恢复数据库。
2. 根据日志，redo这次dump之后已经提交的事务。

## 提前锁释放

普通恢复里，undo通常是physical undo，即将数据项恢复成旧值。但有些高并发操作会提前释放锁，锁释放后数据可能被其他事务修改，physical undo会覆盖后来的合法修改。解决方法为，对undo，使用logical undo执行一个相反的逻辑操作；redo仍为physical redo。

Logical undo中，复杂操作的日志用`<T, O, operation-begin>`和`<T, O, operation-end, U>`包围。其中，O为操作实例的编号，U为logical undo所需的信息，表达相反的逻辑操作。中间为普通物理日志。

崩溃时，从后往前扫描日志。如果遇到operation-end，说明操作已完成，只需要用U执行logical undo。

恢复流程：

1. Redo phase：从最近的checkpoint开始向后扫描，对所有事务redo，使数据库恢复到崩溃瞬间的状态。扫描的同时，建立undo list，遇到start则加入，遇到commit或abort则删除。
2. Undo phase：从日志末尾向前扫描，只处undo list中的事务：
   - 遇到普通物理更新日志，普通undo，并写redo-only log record `<T, X, v_old>`
   - 遇到operation-end，用U做logical undo，然后跳过这个operation内部所有日志，直到operation-begin
   - 遇到redo-only record，忽略
   - 遇到start，将事务从undo list中删除，并写`<T abort>`

## ARIES

ARIES是工业界常用恢复算法。核心思想：

- 用LSN标记日志顺序。LSN（Log Sequence Number）是每条日志都有的递增编号。
- 每个page记录PageLSN。PageLSN表示当前数据页已经反映到哪条日志。Redo某条日志时，如果发现其LSN小于PageLSN，则说明修改已经在数据页上，不用重复redo。
- 恢复时先repeat history，将数据库恢复到崩溃瞬间的状态；再undo 未完成的事务。

Physical redo中，要记录页面的具体变化，如果操作导致page大量移动，则日志很大。ARIES中使用**physiological redo**，即物理指定页、再逻辑表示页内操作。

数据结构：

- LSN：日志的递增编号，类似日志时间戳。
- PrevLSN：每条日志还会记录同一事务的上一条日志的LSN，undo时可沿PrevLSN链走。
- PageLSN：当前page已经包含到哪一条日志的修改。
- RecLSN：对每个页，表示从哪条日志开始可能还没写入磁盘。用于减少redo的扫描范围。
- CLR（compensation log record）：在undo时写的日志，表示撤销操作完成。CLR只会被redo，不会被undo。
- UndoNextLSN：CLR中的字段，表示下一条undo的日志的LSN，即被撤销日志的PrevLSN。
- Dirty Page Table：记录哪些页面是脏页（在内存中被修改，但未写入磁盘），以及每个脏页的RecLSN。

ARIES中使用过fuzzy checkpoint，在checkpoint的log中记录Dirty Page Table、当前活跃的事务列表，和每个活跃事务的LastLSN。脏页在后台慢慢写回。

ARIES恢复过程：

1. **Analysis**：从最近checkpoint开始，设置RedoLSN = min(RecLSN)，即修改过脏页的最早日志。再向后扫描：
   - 遇到新事务，加入undo list
   - 遇到数据更新记录，且page不在dirty page table中，则加入，并设置其RecLSN为更新日志的LSN
   - 遇到事务解释，从undo list中删除
2. **Redo**：从RedoLSN开始向前扫描日志，redo所有事务。过程中跳过不必要的redo：
   - 如果page不在dirty page table中，则跳过
   - 如果日志的LSN小于page的RecLSN，说明日志在数据页变脏前，跳过
   - 如果日志的LSN小于page的PageLSN，跳过
3. **Undo**对undo list中的事务，记录LastLSN，从中选择最大的开始向前：
   - 如果为普通更新，撤销，并写CLR，跳转到该事务的PrevLSN
   - 如果为CLR，不撤销，跳转到CLR的UndoNextLSN
