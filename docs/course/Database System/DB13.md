## 概览

故障的类型：

- 事务故障：逻辑错误，或程序abort。
- 系统崩溃：如断电。通常认为不会破坏磁盘上的非易失性存储内容。
- 磁盘故障：磁盘上数据丢失，通常能检测出来。

恢复算法涉及两部分：

- 事务正常运行时做准备，如日志。
- 故障发生后，根据这些记录把数据库恢复到正确状态。

## 日志

日志记录了数据库修改过程，目的是系统崩溃后还能根据日志判断哪些事务要保留，哪些事务要撤销。一般保存在stable storage（磁盘）上。

一个事务T的日志包括三种：

1. 事务开始时写：`T start`
2. 事务修改数据X之前写：`T X v_old v_new`
3. 事务成功结束时写：`T commit`

## 恢复操作

恢复时有两种可能操作：

- **undo**：使用旧值，撤销未完成事务的修改。
- **redo**：使用新值，重新执行已经提交事务的修改。

Undo和redo都是幂等的，即做一次和做多次效果一样。

日志恢复有两种主要方式：

1. **延迟修改**：事务执行过程中只写日志，不立刻改数据库；等事务提交之后，再根据日志真正更新数据库。崩溃时，如果未提交，则不需要undo；如果已提交，则需要redo。

- 优点：简单
- 缺点：灵活性差

2. **立即修改**：事务还没提交时，它的修改就可以先写到缓冲区，甚至写到磁盘。崩溃时，如果日志有start但没有commit，则说明未完成，需要undo；如果既有start又有commit，则说明已经提交，需要redo。

- 优点：灵活
- 缺点：恢复复杂

## 检查点

Checkpoint避免每次从头扫描日志，加快恢复。

每个checkpoint中完成三件事：

1. 把内存里的日志记录强制写到稳定存储。
2. 把已经修改过的缓冲区块写回磁盘。
3. 在日志中写`checkpoint L`，其中L是checkpoint时仍然活跃的事务列表（即已经开始但还没有commit或abort）。

恢复时，系统从日志末尾往前找最近的`checkpoint L`。找到后，不需要考虑更早的事务，只需要处理L中和checkpoint以后的事务即可。

## 并发事务的恢复

真实数据库中，多个事务并发执行。所有事务共享同一个磁盘缓冲区和同一个日志文件，一个缓冲块里也可能包含多个事务修改过的数据。

下面都假设使用strict two-phase locking（事务修改数据时要拿排他锁，而且这个排他锁通常要持有到事务结束，避免连续回滚）。

两阶段恢复过程：

1. 从日志末尾向前扫描，直到找到最近的 checkpoint。扫描过程中维护两个集合：redo-list 和 undo-list。
2. 确定list元素：
   - 扫描过程中，如果遇到`Ti commit`，则说明Ti已提交，将其加入redo-list；如果遇到`Tj start`，但Tj不在redo-list中，则加入undo-list。
   - L中的事务，如果不在redo-list中，则加入undo-list。
3. 执行恢复操作：
   - 对undo-list中的事务，从日志末尾向前扫描，遇到它们的修改就用旧值撤销。
   - 对redo-list中的事务，从checkpoint开始向前扫描到日志末尾，遇到它们的修改就用新值重做。

## 日志缓冲

一条日志记录通常很小，而磁盘写入单位通常是块，所以日志记录也会先缓存在内存中，等日志块满了或需要强制写日志时，再一起写到stable storage。

**Write-Ahead Logging**（WAL，先写日志规则）：必须先将日志写到稳定存储，再执行对应的操作，如将数据块写回磁盘、进入commit状态等。

两个策略：

- **no-force**：事务提交时，不要求立刻把它修改过的所有数据块写回磁盘。该策略下commit快，但如果崩溃时提交事务的数据还没落盘，则需要redo。
- **steal**：缓冲区不够用时，可以把包含未提交事务修改的数据块提前写回磁盘。该策略下buffer管理更灵活，但如果之后事务回滚或系统崩溃，则需要undo。

另外，**latch**为一种很短时间持有的内部锁，用来保护内存中的数据块。Eg，buffer block写回磁盘时，需要先拿到这个块的exclusive latch，确保没有事务正在修改它；然后flush相关日志；再把数据块output到磁盘；最后释放latch。

## 非易失性存储损坏

如果发生Failure with Loss of Nonvolatile Storage（非易失性存储损坏），如磁盘损坏，则可能日志和数据库都丢失。解决方法为，定期把整个数据库复制到更可靠的stable storage，这一操作称为**dump**。

Dump流程：

1. 把内存中的日志写到稳定存储，把buffer中的数据块写回磁盘。
2. 复制整个数据库。
3. 在日志中写入`dump`。

恢复流程：

1. 从最近一次dump恢复数据库。
2. 用日志redo这次dump之后已经提交的事务。

## 提前锁释放

普通恢复里，undo通常是physical undo，即将数据项恢复成旧值。但有些高并发操作会提前释放锁，锁释放后数据可能被其他事务修改，physical undo会覆盖后来的合法修改。

解决方法为使用logical undo，执行一个相反的逻辑操作。

## ARIES

核心思想：

- Write-Ahead Logging
- Repeating History：恢复时先把数据库重做到崩溃瞬间的状态，包括已提交事务和未提交事务的修改；然后再撤销未完成事务。
- Logging Changes During Undo

数据结构：

- LSN（Log Sequence Number）：指每条日志的递增的编号。
- PageLSN：对每个数据页，表示其映射到哪条日志。Redo时，如果发现页面的PageLSN已经大于等于某条日志的LSN，就说明这条修改已经在页面上，不需要重复redo。
- RecLSN：对每个页，表示从哪条日志开始可能还没写入磁盘。用于减少redo的扫描范围。
- Dirty Page Table：记录哪些页面是脏页，以及每个脏页的 RecLSN。

ARIES恢复过程：

1. 第一遍Analysis：从最近checkpoint开始分析，找出崩溃时哪些事务没完成、哪些页面是脏页、redo应该从哪里开始。
2. 第二遍Redo：从RedoLSN开始向前扫描日志，重复历史，但会利用Dirty Page Table和PageLSN跳过不必要的redo。
3. 第三遍Undo：把未完成事务倒着撤销。
