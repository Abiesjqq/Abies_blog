## 实体与联系

### 联系集

几个概念：

- **实体（entity）**：一个现实世界中存在的、能和其他对象区分开的对象。实体集（entity set）是有相同属性的实体的集合。esp.，在数据表中，每一行是一个entity，每一列是一个attribute，整个表是一个entity set。
- **联系（relationship）**： 若干实体之间的关联。联系集（relationship set）是同类型联系的集合。
- **属性（attribute）**： 描述实体或联系的某些性质。可以分为以下几种（只是作为分析问题，SQL中一定为原子属性）：
  1. 简单属性（原子性的属性）、复合属性（可继续拆分）
  2. 单值属性（一个实体对应一个值）、多值属性（一个实体可对应多个值）
  3. 基属性、派生属性（能由基属性推导得出）

- **联系的度（degree）**：参与一个联系集的实体集的个数。esp.，度为3时为三元联系，表示“A在不同的B下有不同的C”这类关系。

映射基数和参与约束分别表示参与数量的上限和下限：

**映射基数（mapping cardinality）**：指一个实体可以和另外多少个实体相联系的数量上限。分为一对一、一对多、多对一、多对多这四种情况。

**参与约束（participation）**：用于约束最少参与的数量。分为以下两种情况：

1. 全参与：每个实体都至少参与一次这个联系。用双线表示。
2. 部分参与：有些实体可以不参与该联系。

**_Q：怎么确定relationship set的super key？_**

和联系集的cardinality有关：

- 一对多时，“多”的一方为super key。
- 多对多时，super key为两者主键的结合。

### 特殊的联系集

Recursive relationship set（自环联系集）用于表示一个实体集中，一部分实体与另一部分实体间的关系（如manager和worker的work-for关系）。其中，不同类的实体称为不同role。

多元联系指一个关系涉及到多个实体集。多元联系一定能转化为二元联系，转化方法为：新增实体集E，将原多个实体集之间的关系，转化为分别与E的关系。

### 弱实体集

**弱实体集（weak entity set）：** 没有super key的实体集。即它不能只靠自己的属性唯一标识自己，而要依靠其他实体集（表）。

弱实体集的存在依赖于一个强实体集，这个强实体集叫属主实体集（owner entity set）或标识实体集（identifying entity set）。弱实体集和其属主的联系称为标识性联系（identifying relationship）。强弱实体集之间的关系一定是一（强）对多（弱）关系。

**分辨符（discriminator）/部分码（partial key）：** 在弱实体集中，某些属性可用于在同一个强实体集中区分不同的弱实体集，这些属性集称为分辨符或部分码。

**_Q：弱实体集的super key是什么？_**

弱实体集的super key = 强实体集的super key + 弱实体集自身的discriminator。

设计ER图时，强实体集的键不存在弱实体集中，否则造成冗余；具体用table表示时，必须和属主实体集中的主键合并成一张表，不用另外用一张表表示强弱间的relationship。

## 扩展ER特性

### 特殊化与泛化

**特殊化（specialization）** 指从一个“大类”向下细分成更具体的“小类”，**泛化（generalization）** 指将若干具有共同特征的实体集合并成一个更高层实体集。在ER图中，两者都用`ISA`倒三角形表示。

按照“一个上层实体能不能同时属于多个下层实体集（横向关系）”，分为以下两种：

1. disjoint（不相交）：一个实体至多属于一个下层实体集。
2. overlapping（重叠）：一个实体可同时属于多个下层实体集。

按照“一个上层实体是不是至少属于一个下层实体集（纵向关系）”，分为以下两种：

1. total（完全泛化）：上层实体中的每个实体都必须至少属于一个下层实体集。
2. partial（部分泛化）：上层实体中的实体，不一定必须属于某个下层实体集。

### 聚合

当部分实体集同时包含在两个多元联系时，会产生冗余。

**聚合（aggregation）**：允许将关系本身视为“实体集”，去参加新的联系。

Cardinality的表示：

- 箭头表示最多一个，直线表示可以多个

Participation的表示：

- 双线表示“全参与”，单线表示“部分参与”
- 或联合表达：用`A..B`，其中A表示最少、B表示最多。B可用用`*`表示上限不限。

自环联系集的表示：

- 关系与实体集连接两次，在线上可标role的名称。

泛化的表示：

- 用`ISA`的倒三角表示，三角形上面是上层实体集，下面连接各个下层实体集，三角形旁写约束。
- ISA下方写`disjoint`或`overlapping`，表示子类能否重叠。
- ISA上方单线表示部分泛化，双线表示完全泛化

## 数据表转化

下面考虑从ER模型转化到数据表。

- 复合属性：将其中每个属性作为单独的属性存储。
- 多值属性：不同组合的属性存储在不同的表中。
- 弱实体集：将属主实体集的primary key和弱实体集的属性拼成一张表。ER图中联系强弱实体集的关系不对应任何表。
- 关系：将由实体集产生的super key，和关系本身的属性拼成一张表。
  
泛化关系的存储：

1. 方法一：公共属性存成一张表R；各自属性加R的primary key，存成另一张表。缺点为查询时需访问两张表。
2. 方法二：对每个子类的所有属性，单独存一张表。缺点为公共属性冗余。

数据表合并：

- 对于“一对多”的关系，可以合并到“多”的表中。但如果“多”端为部分参与，合并会导致产生null值。
- 对于“一对一”的关系，可以合并到任一端的表中。
