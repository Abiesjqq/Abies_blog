## 查询处理概览

总体问题：数据库拿到一条SQL后，怎样把它变成真正能执行的操作，并尽量执行得快。

SQL执行大致为以下几步：

1. 语法分析与翻译：将SQL翻译成扩展关系代数（ERA）。
2. 查询优化：在等价的表达式中，选择代价最低的执行方案。
3. 执行：这一步有不同做法。

## 查询代价

查询代价估计简化为block transfer（block的读取/写入）和seek（磁盘寻道）。

符号规定：block transfer次数记为 $b$，单次时间记为 $t_T$；seek次数记为 $S$，单次时间记为 $t_S$。$t_S$ 远大于 $t_T$，因此需要尽量减少随机读取。

Buffer的影响：内存越大，越少访问磁盘，查询代价越小。

## 选择操作

### 查询方法

- **Linear Search**

逐个读取block，检查每条记录。线性扫描是最通用的方法。

记r表的block数为 $b_r$，时间为

$$
b_r t_T + t_S
$$

如果查询的是primary key，由于主键只出现一次，平均只需扫描半张表，transfer次数为 $b_r / 2$。

- **Binary Search**

当文件按查询的属性排序、且block连续存储时，可以使用二分查找。二分查找不常用，因为更多使用索引。

时间复杂度：

$$
\lceil \log_2 b_r \rceil t_T + \lceil \log_2 b_r \rceil t_S
$$

- **Bitmap Index Scan**

PostgreSQL中常用位图索引扫描。先用bitmap记录哪些page中可能有记录，再读取为1（可能有）的页。

### 查询类型

- **主索引+主键等值查询**

用主索引进行主键的等值查询时，主索引保证有B+树，主键保证最多匹配一条记录，故直接在B+树上查找。

记 $h_i$ 为B+树高度，$h_i(t_T + t_S)$ 为树上查找时间，$1(t_T + t_S)$ 为由叶节点指针访问真实数据块的时间，总时间为

$$
(h_i + 1)(t_T + t_S)
$$

- **主索引+非主键等值查询**

主索引保证有B+树，但非主键导致可能匹配多条记录，这些记录通常连续存储，因此找到第一条后顺序扫描即可。

记 $b$ 为包含所有匹配记录的数据块数。B+树上查找时间为 $h_i (t_T + t_S)$，由叶节点访问真实数据块时间为 $t_S$，顺序读取数据块时间为 $bt_T$，总时间为

$$
h_i (t_T + t_S) + t_S + bt_T
$$

- **辅助索引+非主键等值查询**

由于辅助索引不连续分布，不能顺序扫描。在B+树叶节点中找到所有对应的索引项后，依次访问真实数据块。

记 $n$ 为满足条件的记录数，总时间为

$$
(h_i + n)(t_T + t_S)
$$

- **主索引+范围查询**

对 $\ge$ 类型，用B+树找到第一个满足的叶节点指针，访问真实数据块，再顺序扫描；对 $\le$ 类型，由于数据文件本身按从小到大顺序排列，从开头顺序扫描，直到遇到第一个不满足的数据为止。

- **辅助索引+范围查询**

对 $\ge$ 类型，用B+树找到第一个满足的叶节点指针，沿B+树叶节点顺序扫描索引项，再根据每个指针访问真实数据；对 $\le$ 类型，从B+树叶节点最左侧开始扫描。

- **AND条件查询**

两种方法：

1. 先查返回结果最少的索引，再在返回的记录中检查其他索引。
2. 使用复合索引。
3. 分别查多个索引，对返回结果取交集。

- **OR条件查询**

分别查多个索引，对返回结果取并集。如果某个条件没有索引，则直接全表扫描。

## 排序

外部归并排序，略。总代价为

$$
b_r \left( 2\lceil \log_{M-1}\left( \frac{b_r}{M} \right)\rceil +1 \right)
$$

## 连接操作

规定r表为外表、s表为内表。记r表中记录条数为 $n_r$，block数为 $b_r$；s表中记录条数为 $n_s$，block数为 $b_s$。

### Nested-Loop Join

将r表中每一条记录与s表中记录比较。

```py
for each tuple tr in r:  # read block
    for each tuple ts in s:  # read block
        if tr 和 ts 满足连接条件:
            输出
```

磁盘代价：假设内存中只能放r表、s表各自一个block。首先需读取r中的所有block，代价为 $b_r$；对r每条记录，都要重新扫描整个s的block，代价为 $n_r \times b_s$。总代价为

$$
n_r \times b_s + b_r
$$

### Block Nested-Loop Join

每次将r的一个block读入内存，然后让这个block中所有tuple都和s比较。由此减少s表扫描次数。

由于外层表决定了内层表被重复扫描多少次，应选择较小的表作为外表。

```py
for each block Br in r:  # read block
    for each block Bs in s:  # read block

        for each tuple tr in Br:  # no reading
            for each tuple ts in Bs:  # no reading
                if tr 和 ts 满足连接条件:
                    输出
```

磁盘代价为：

$$
b_r \times b_s + b_r
$$

### Indexed Nested-Loop Join

在nested-loop join的基础上，将内表从全表扫描改成索引查找，即用外表记录的join key，在内表的索引中查找匹配tuple。索引查找可使用B+树，因此比nested-loop更快。

```py
for each tuple tr in r:  # read block
    v = tr.A  # get join key
    use index on s.B to find all tuples ts where ts.B = v  # search time on B+ read
    for each matched tuple ts:
        if tr.A = ts.B:
            输出 tr ⋈ ts
```

记一次索引查找的成本为 $c$，则总时间为：

$$
b_r + n_r \times c
$$

### Merge Join

当两表都按join key排序、等值连接时，使用双指针。总时间为：

$$
b_r + b_s
$$

Hybrid merge join：一个关系已经排序，另一个关系虽然没有排序，但它的连接属性上有secondary B+ tree index，仍能对B+树叶节点有序扫描，完成merge。

### Hash Join

适用于等值连接。若两tuple的join key相等，那么它们经过同一个hash函数后，一定会进入同一个分区。利用此原理，先用相同的hash函数将两表分成若干bucket，只需对bucket中分别等值连接，减小比较范围。

Hash join通常分成两阶段：

1. Partition phase（分区阶段）：将r、s用同一个hash函数分区。
2. Build + probe phase（探测阶段）：对每个bucket编号 $i$，将较小分区 $s_i$ 读入内存，在内存中为 $s_i$ 建立hash table；再扫描 $r_i$，对 $r_i$ 中每条tuple，用join key和内存中的hash table查找匹配项；输出连接结果。

如果分区数量大于内存页数，或者某些partition仍然过大，就可能需要递归分区（recursive partitioning）。对太大的partition，继续用另一个hash函数分区。

常见问题与解决方法：

- Skew：partition大小不均匀。解决方法为选择更好的hash函数。
- Overflow：可能因为有大量重复key，也可能因为hash函数不均匀。解决方法为增大分区数、递归分区、或退化为block nested-loop join。

### Complex Join

复杂连接指连接条件不只是一个等值条件，分为多条件AND（conjunctive join）和多条件OR（disjunctive join）等。

- **Conjunctive join**

1. 对每对tuple，同时检查所有条件。效率低。
2. 先执行成本较低的join条件，再检查剩余条件。第一次join能大量减少中间结果。

- **Disjunctive join**

分别计算每个join的结果，再取并集。由于分别join时可能产生重复结果，union时需要去重。
