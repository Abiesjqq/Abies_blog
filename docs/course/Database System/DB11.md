## Transection概念

**_Q：为什么需要事务？_**

数据库系统处理多用户并发执行、系统故障时，易丢失数据。需要一种机制，把一组操作看成一个整体，即事务（transaction）。

**_Q：事务是什么？_**

Transaction是程序执行的逻辑工作单元，通常由多条SQL语句组成，由commit或rollback结束。

事务的特性：ACID

- Atomicity（原子性）：一个事务中的所有操作，要么全部成功，要么全部不生效。
- Consistency（一致性）：事务执行前后数据库都是正确的。
- Isolation（隔离性）：多个事务并发执行时，每个事务感觉像是在独立运行。即数据库要隐藏事务的中间状态，不能让别的事务随便看到未完成的数据。
- Durability（持久性）：事务一旦提交，它对数据库的修改就必须永久保存，即使之后系统崩溃、断电，提交过的数据也不能丢失。

四个性质实现：

| 性质               | 主要由谁保证                  |
| ------------------ | ----------------------------- |
| 原子性 Atomicity   | 恢复机制 / 日志               |
| 持久性 Durability  | 恢复机制 / 日志 / 磁盘写入    |
| 隔离性 Isolation   | 并发控制                      |
| 一致性 Consistency | 用户逻辑 + 数据库约束共同保证 |

事务可能的状态：

| 状态                | 含义                                               |
| ------------------- | -------------------------------------------------- |
| Active              | 事务正在执行                                       |
| Partially committed | 最后一条语句已经执行完，但结果可能还没真正写入磁盘 |
| Failed              | 事务发现无法继续执行                               |
| Aborted             | 事务已回滚，数据库恢复到事务开始前                 |
| Committed           | 事务成功提交                                       |

**_Q：怎么实现Atomicity/Durability？_**

一个简单的方法为，使用shadow database。事务在拷贝后的数据库上操作。如果失败，则删除新数据库，保证atomicity；如果成功，则将新数据库flush写入磁盘，再原子修改数据库指针，保证持久性。

但这种方法每次要复制数据库，低效且不支持并行。实际使用日志。

## 并发执行与调度

**_Q：调度（schedule）是什么？_**

数据库中多个事务同时运行，但单核CPU每次只能执行一个指令，因此需要线性排序，即形成schedule。Schedule本质是多个事务操作的交错执行序列，要求不能丢操作、保持事务内部顺序。

两种调度形式：

1. Serial schedule（串行调度）：先后执行不同transaction，执行顺序不同结果可能不同。

- 优点：一定正确
- 缺点：没有并发，性能差

2. Concurrent schedule（并行调度）：不同transaction交叉执行。调度错误会导致结果错误。

- 优点：效率高
- 缺点：可能出错

### Serializability

真正影响并发正确性的只有读写数据库，下面将事务全部简化为read和write。

**_Q：怎么判断并发的调度是否正确？_**

正确的并发调度一定等价于某种串行调度，如果满足，则称其为serializable。按等价关系，分为以下两种：

- Conflict serializable：

当两个事务访问同一数据，且至少一个是写时，会导致冲突（conflict）。冲突操作不能交换顺序，因此决定了时间的先后关系。

如果一个schedule能通过若干次交换不冲突操作，得到另一schedule，则称二者为conflict equivalent。进一步，如果和某个串行调度conflict equivalent，则称该调度为conflict serializable。

- View serializability：

有些事务无法通过交换非冲突操作变成串行，但最终效果仍然一致。E.g.，blind write（没有读、直接写）。

若当前调度与某个串行调度所有读到的数据来源一致，即读初值一致、写读关系一致、最后写的事务一致，则称为view serializability。

View serializability的判定为NP-complete，使用较少。

**_Q：怎么判断调度是否为conflict serializable？_**

使用precedence graph。点表示事务；边表示conflict，从冲突中先进行的事务指向后进行的事务。一个schedule是conflict serializable，当前仅当precedence graph无环。

如果无环，则可以拓扑排序（将有向无环图中所有节点排成线性序列，且保持边的先后关系。Kahn算法，根据入度构建队列）

## Recoverability

**_Q：如果事务失败，怎么保证已运行过的其他事务不被污染？_**

可恢复调度（recoverable schedule）要求，如果事务A读了事务B写过的数据，则B一定比A先提交。否则，称读到的、未提交的数据为dirty data，对应的读为dirty read。

如果一个事务失败，其他读的事务都要回滚，即cascading rollback。级联回滚代价大，需要尽量避免，因此加强可恢复调度的条件：

无级联调度（cascadeless schedule）要求，如果事务A读了事务B写过的数据，则B的提交一定在A读之前。即提交前不允许其他事务读。
