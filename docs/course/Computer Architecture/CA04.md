## 虚拟内存原理

**_Q：Virtual memory是什么，为什么需要？_**

虚拟存储器将主存+磁盘组织成“逻辑内存空间”，程序看到自己拥有一块连续的、很大的内存空间，但实际程序的数据可能分散放在物理内存的不同位置，甚至有一部分暂时放在磁盘上。

用处：

- 提高内存利用率：不需要每个进程连续物理内存
- 支持多个程序同时运行：多进程各自认为自己有完整空间
- 实现内存保护：物理地址隔离
- 扩展可用内存容量：部分实际页可在磁盘中

使用VM后，一次读取地址需要访问两次主存：一次访问页表（页表也存在主存中），得到物理地址；一次访问物理地址。

**_Q：VM和cache的关系？_**

- Cache解决速度问题，VM解决容量、保护、多进程管理等问题。
- 都利用locality和分块思想。
- VM访问更慢、miss penalty更大，但hit rate更高。
- TLB miss时需要查询内存，往往先查询cache。

## 虚拟内存的实现

### 分页、分段与地址翻译

**_Q：怎么管理虚拟内存？_**

按内存分块的大小是否固定，分为以下两种：

1. **分页（paging）**：将内存切成固定大小的virtual page，常用。虚拟地址拆成页号（page number）和页内偏移两部分，即`PPN+offset`。由于虚拟和物理页大小相同，offset不需要翻译。

- 优点：容易管理，无外部碎片
- 缺点：内部碎片（可能页内部分空间用不到）

2. **分段（segmentation）**：按逻辑意义划分为大小不固定的段。虚拟地址分为段号和段内偏移两部分，即`#seg+offset`。

- 优点：更符合程序逻辑
- 缺点：外部碎片（空闲区域碎片化）；替换时更困难，因为要找到连续的、变长的空闲内存片段。

### 页表与TLB

**_Q：什么时候用到虚拟地址翻译？_**

1. 取指访存：根据PC取指时，将PC虚拟地址翻译为PC物理地址，再取指。
2. 数据访存：load/store指令读写时，要访问数据的虚拟地址翻译为物理地址。

**_Q：CPU怎么将虚拟地址翻译成物理地址？_**

即，怎么根据CPU给出的虚拟地址（VA）得到物理地址（PA）？

1. 虚拟地址拆分：VA=VPN+offset，其中VPN（virtual page number）为虚拟页号。
2. 页表查询：PTE（page table entry，页表项）记录VPN到PPN（physical page number）的映射关系，通过VPN和页表得到PPN。
3. 物理地址拼接：PA=PPN+offset。

Lab4中，为解决单级页表过大的问题，设计为二级页表。将VPN拆为等长的VPN1和VPN2两部分，分别对应L1和L2页表。L1页表中，若有数据，则存储结果和0拼接得到指向对应L2页表的指针；若无数据，则存储空指针（空指针节省大量空间）。

**_Q：怎么通过物理地址（PA）得到页表数据？_**

PPN为物理内存页（physical frame）的编号；若每个页表大小为 $N$，则物理页的起始地址为 $N\times PPN$。

若对应的物理页无效，则发生page fault，说明该页不在物理内存中、而在磁盘中。操作系统介入，将该页从磁盘/SSD调入某个physical frame，再更新页表：

- 若调入空闲frame，则设置页表中该VPN对应为新PPN；
- 若替换frame，则设置页表中该VPN对应为新PPN，将新PPN原来的PTE设为invalid。替换使用LRU策略。

page fault（virtual memory miss）在两种情况下发生：

1. 页不在主存中（valid=0），可能第一次访问时磁盘页未加载到主存中，也可能被替换
2. 物理页无访问权限

在main memory中，frame可视为按fully associative放置，即一个virtual page可以放到任意空闲的frame中，只要建立VPN-PPN的映射即可。因为主存中的miss即page fault，代价大，而fully associative下miss rate低。虚拟内存之所以能使空间看起来更大，因为程序只需要提供虚拟地址，但对应的物理页可能在内存中、也可能在磁盘中，且顺序可能打乱。

**_Q：Page size对性能有什么影响？_**

- Page size大：页数少，page table小，占用内存少；TLB更大，但命中率增加；每次transfer时间长，但能调换更多的数据。
- Page size小：减少内部碎片，节省storage。

现代系统往往支持多页面大小（multiple page sizes）。

**_Q：怎么加快页表查询速度？_**

TLB（translation lookaside table）为page table的cache。TLB entry的内容为：

- tag：VPN
- data：PPN，valid bit，dirty bit，use bit（是否最近访问，用于LRU替换），protection bit（权限控制）

使用以下策略：

- 使用全相联，每条VPN在全体TLB中查找：因为miss penalty较大，全相联减少冲突。
- 冲突时使用LRU替换。
- 使用write back+write allocate策略：因为写入主存/磁盘耗时长
