## 动态调度概念

**_Q：为什么需要乱序执行？_**

顺序流水线中，流水线会被慢指令占用，导致后面快指令一直等待。动态调度的核心思想为，能执行的指令先执行，即指令级并行（ILP，Instruction-Level Parallelism）。此时指令流入和流出流水线的顺序可能不同。

乱序执行带来新的data hazard：

- WAR：实际为先读后写（称为anti-dependence，实际可以写到其他寄存器），但乱序后写指令提前到读之前。
- WAW：两条写指令，第二条的目标寄存器为第一条的源寄存器，乱序下先后顺序改变。

寄存器的重命名能在一定程度上解决WAR和WAW。

### Scoreboard算法

Scoreboard算法中，将ID阶段拆成两阶段，整体流水线为IS-RO-EX-WB：

1. IS（issue）：用于发射指令，检查structure hazard。
2. RO（read operands）：用于等待操作数准备好。检查data hazard，如果操作数未准备好则等待。

硬件结构上，新增**scoreboard**（计分板）和**functional unit**（功能部件，一个加法、一个除法、两个乘法、一个立即数计算部件）。Scoreboard记录各指令、功能部分的状态，用于调度，包含以下三张表：

1. **指令状态表**：对每条指令，记录目前执行到哪里。

| Instruction | IS  | RO  | EX  | WB  |
| ----------- | --- | --- | --- | --- |
| FLD         | √   | √   | √   | √   |
| MUL         | √   |     |     |     |

2. **功能部件状态表**：对每个功能部件，记录当前状态，包含：

- Busy：功能部件是否被占用
- Op：操作类型
- Fi：目的寄存器
- Fj/Fk：源寄存器
- Qj/Qk：源操作数将来由哪个功能部件产生，完成计算后记为null。
- Rj/Rk：操作数是否还需要从寄存器中读取，完成计算后记为no。

| FU      | Busy | Fi  | Fj  | Fk  | Qj  | Qk  | Rj  | Rk  |
| ------- | ---- | --- | --- | --- | --- | --- | --- | --- |
| Integer |      |     |     |     |     |     |     |     |
| Mult1   |      |     |     |     |     |     |     |     |
| Mult2   |      |     |     |     |     |     |     |     |
| Add     |      |     |     |     |     |     |     |     |
| Divide  |      |     |     |     |     |     |     |     |

实际还包括load和store部件。

对于源操作数，可能依次有三种状态：

- R=no，Q!=null：对应的功能部件还未计算完
- R=yes，Q=null：功能部件计算完，可以直接读取
- R=no，Q=null：已完成读取

3. **结果寄存器表**：对每个寄存器，记录哪个功能部件会将结果写入。

| Register | Qi    |
| -------- | ----- |
| F0       | Mult1 |
| F8       | Add   |

**_Q：Scoreboard算法怎么处理RAW？_**

如果操作数未准备完成，则停在RO阶段。

**_Q：Scoreboard算法怎么处理WAR/WAW？_**

如果需要旧值的指令未完成读，则阻止WB。该处等待为scoreboard算法的主要缺点。

**_Q：Scoreboard算法有什么问题？_**

1. 检测到name dependence后等待，而不主动解决（如重命名）
2. 未按照程序原来顺序执行，可能导致非精确异常。

??? examples "例子"

    设不同类型指令的执行时间为：

    - LD 1 cycle
    - ADD 2 cycles
    - MUL 10 cycles
    - DIV 40 cycles

    指令序列为：

    ```
    FLD     F6, 34(R2)
    FLD     F2, 45(R3)
    FMUL.D  F0, F2, F4
    FSUB.D  F8, F6, F2
    FDIV.D  F10, F0, F6
    FADD.D  F6, F8, F2
    ```

    |      | IS  | RO  | EX    | WB  |
    | ---- | --- | --- | ----- | --- |
    | FLD  | 1   | 2   | 3     | 4   |
    | FLD  | 5   | 6   | 7     | 8   |
    | FMUL | 6   | 9   | 10-19 | 20  |
    | FSUB | 7   | 9   | 10-11 | 12  |
    | FDIV | 8   | 21  | 22-61 | 62  |
    | FADD | 13  | 14  | 15-16 | 22  |

    注意点：

    - 示例中没有load buffer，两条load指令需先后执行
    - IS、RO在上一条的WB的下一拍
    - 如果寄存器作为源寄存器的指令未完成RO，则作为目标寄存器的指令不进行WB
    - IS阶段不乱序

    RAW中，功能部件状态表的更新为（以7-9拍为例）：

    - 第7拍：mul.Rj=no, Qj=load，表示mul源操作数由load产生，但未计算完成
    - 第8拍：mul.Rj=yes, Qj=null，表示mul的源操作数计算完成，但未读取
    - 第9拍：mul.Rj=no, Qj=null，表示mul已完成读取

### Tomasulo算法

在scoreboard算法基础上引入重命名。进入流水线后，如果数据已准备好，则直接记录实际值，而不是寄存器名称；如果未准备好，则记录将由哪里产生。

指令流出逻辑（从指令队列到保留站/buffer）：保留站/buffer有位置。

![alt text](assets/Tomasulo.png)

硬件结构上，增加：

1. **Reservation station**：位于功能部件前（图中绿色部分）。所有指令顺序发射后，检查RS是否有容量，有则进入、自动重命名、等待操作数准备好和功能部件空闲。
2. **CDB**（公共数据总线）：在WB阶段，将指令执行结果广播到所有需要的位置（图中蓝线），实现各寄存器同时得到结果、自动唤醒。总线只用于传递数据，不监听。

数据表包括：

1. **指令状态表**：只包括IS、EX、WB三个阶段。因为RO由reservation station机制代替，源操作数计算完成后，直接写入Vj/Vk；如果未计算完，记录Qj/Qk，计算完后通过CDB发送数据。

2. **保留站状态表**：类似scoreboard中的功能部件状态，对每个功能部件记录（删除Fi/Fj/Fk，因为不记录具体寄存器名称）：

- Busy：该部件是否被占用
- Vj/Vk：两个源操作数的数值
- Qj/Qk：未来产生两个源操作数的功能部件。完成计算后记为null。
- A：只用于load/store部件，存放计算后的地址

3. **寄存器状态表**：同scoreboard中的寄存器状态表。

??? examples "例子"

    指令和执行时间同上。

    |                  | IS  | EX    | WB  |
    | ---------------- | --- | ----- | --- |
    | FLD F6, 34(R2)   | 1   | 2-3   | 4   |
    | FLD F2, 45(R3)   | 2   | 3-4   | 5   |
    | FMUL F0, F2, F4  | 3   | 6-15  | 16  |
    | FSUB F8, F6, F2  | 4   | 6-7   | 8   |
    | FDIV F10, F0, F6 | 5   | 17-56 | 57  |
    | FADD F6, F8, F2  | 6   | 9-10  | 11  |

    注意点：

    - IS拍数指读入拍数，不是所有源操作数准备完成的拍数
    - 若寄存器同时作为源寄存器和目标寄存器，源寄存器对应指令IS完成后，目标寄存器对应指令就可以WB

### Hardware-Based Speculation

Tomasulo算法中，指令乱序流出，导致异常难处理、分支预测难恢复等问题。Hardware-based speculation中加入**ROB**（reorder buffer），在指令执行完后排序等待，保证顺序流出、精确异常。

ROB增加在CDB后。指令进入流水线时（IS阶段），分配ROB编号，记录在reservation station table中；指令执行完后，由CDB写入ROB、而不是直接写寄存器，按编号顺序依次commit时才更新架构寄存器。

![alt text](assets/Hardware_Based_Speculation.png)

Reorder buffer代替之前的指令状态表，对每个ROB编号（由循环队列维护），记录：

- Busy：该编号是否被占用
- Instruction：对应的指令
- Status：当前执行到的阶段，分为IS、EX、WB、Commit四阶段
- Object：目标寄存器
- Value：仅在WB、Commit阶段记录，表示指令执行后最终结果。写入Value的同时，也写入Reservation station table中依赖的指令的Vj/Vk。

Register status table的修改：

- 对每个寄存器，不再记录产生器的功能部件，而是记录ROB编号。

Reservation station table的修改：

- 增加Dest项，记录该指令的ROB编号。指令执行完后，先写到ROB中对应位置，到Commit阶段再真正写入目标寄存器。
- Qj/Qk不再记录功能部件，而是记录产生源操作数指令的ROB编号。

??? examples "例子"

    指令和执行时间同上。

    |                  | IS  | EX    | WB  | commit |
    | ---------------- | --- | ----- | --- | ------ |
    | FLD F6, 34(R2)   | 1   | 2-3   | 4   | 5      |
    | FLD F2, 45(R3)   | 2   | 3-4   | 5   | 6      |
    | FMUL F0, F2, F4  | 3   | 6-15  | 16  | 17     |
    | FSUB F8, F6, F2  | 4   | 6-7   | 8   | 18     |
    | FDIV F10, F0, F6 | 5   | 17-56 | 57  | 58     |
    | FADD F6, F8, F2  | 6   | 9-10  | 11  | 59     |

    注意点：

    - IS和commit一定顺序，即后面指令的拍数严格大于前面
    - 依赖的两条指令，前一条WB后，后一条即可EX，而不用等到前一条commit

## 多发射处理器

多发射处理器指每个时钟周期可以发射多条指令，其中“发射”指把已经取出、译码后的指令送到对应的功能部件中执行。分为以下几种：

### Superscalar

Superscalar（超标量处理器）每个周期发射的指令条数不固定，硬件或编译器尽量在每个周期找出多条可以并行执行的指令。如果上限是n，则称为n-issue processor。

Superscalar的实现思路有：

1. **静态调度**：编译器在生成机器码时，尽量将能并行执行的指令排在一起，使处理器更容易同时发射。
2. **动态调度**：使用前面的算法，由硬件在运行时决定。

#### 静态调度的多发射技术

对于n-issue处理器，取指阶段一次取n条指令，组成issue packet。处理器检查这个包中的指令能不能一起发射，即检查包内部、包和当前正在执行指令的structural hazard和data hazard。无冲突的指令发射，有冲突的指令暂停。

一般整数指令和浮点指令分开操作、使用不同的功能部件，为了简化通常将2-issue设计成“一条整数+一条浮点”的结构。

#### 动态调度的多发射技术

在“一条整数+一条浮点”的模型中，需要顺序发射到RS，乱序EX、WB，否则（乱序发射）可能导致寄存器重命名关系错乱。

可以将表结构拆开，整数表和浮点表分离。同一周期内，整数指令和浮点指令可以分别进入自己的保留站。

使用硬件推测时，即使分支还没有真正完成，只要预测了分支方向，就允许后续指令先执行；但是执行结果不能马上提交到正式寄存器或内存，而是先放到ROB中。如果预测正确，则按顺序commit；如果预测错误，则清空错误路径上的ROB项，恢复状态。

### VLIM

VLIW（Very Long Instruction Word，超长指令字）指把多条可以并行执行的操作，提前组合成一条很长的指令，每个位置叫一个operation slot。VLIW的每个字段可以直接、独立地控制一个功能单元。

VLIWM（Very Long Instruction Word Machine，超长指令字处理器）每个周期发射的操作数固定。由编译器判断能否并行，打包成VLIW；如果某个槽找不到合适的操作填，则填NOP。

VLIW 常和循环展开一起使用，即将多轮循环统一展开，以暴露更多 ILP。

缺点：

- 程序代码长度增加
- 操作槽不能总是填满
- Lockstep mechanism：中一个操作因某些原因暂停，整个处理器可能都要暂停。
- 机器码兼容性差：整数槽、浮点槽、load/store槽个数固定

## Superpipelining

Superpipelining为把一个流水线阶段切得更细，让流水线周期变短，每一段执行更小的功能。E.g.，MIPS R4000的8级流水线。
