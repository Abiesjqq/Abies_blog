## MIMD概览

并行的趋势：ILP --> TLP --> MIMD。TLP意味着多个程序计数器，因此通常通过MIMD实现。

两种MIMD架构：

- **Multi-processor system**：系统中只有一个统一的地址空间，所有处理器共享。但物理实现上，可以物理分布、逻辑共享（NUMA）。
- **Multi-computer system**：每个处理器都有私有内存（local memory/private memory），其他处理器不能直接访问，而是通过网络发送消息。通常使用NORMA（no-remote memory access）模型，即每个节点只能直接访问本地内存，远程数据需要massage passing。

## Multi-Processor System

下面所有方法均共享内存，区别在于处理器访问不同内存位置的时间是否相同，以及数据是放在普通内存中还是以缓存形式迁移。

并行处理的挑战在于：

- 程序中可并行的部分有限：当有n个处理器时，理论speedup为n；但考虑可并行指令占比为k（k<1），则实际speedup为

$$
\text{speedup} = \frac{1}{(1-k)+\frac{k}{n}}
$$

- 处理器之间通信有成本：对于remote memory访问，访问时暂停其他处理器，产生通信开销。此时，总CPI为base CPI+远程访问指令的比例乘带来的周期数延迟。

### UMA

UMA（uniform memory access，统一内存访问）的物理内存为集中式。核心特征为一致性访问，即任何处理器访问任意内存单元的时间相同。也称为Symmetric Multiprocessor。

处理器通过ICN与shared memory连接，ICN可以是bus、crossbar或。每个处理器可以加各自的cache或memory。

优点为模型简单；缺点为可扩展性差，当处理器数量增加时，总线和内存会变成瓶颈。

### NUMA

NUMA（non-uniform memory access，非统一内存访问）的物理内存为分布式。核心特征为非一致性访问，即所有处理器共享一个统一地址空间，但访问本地内存快，访问远程内存慢（因为访问remote memory需要load/store）。

每个处理器都有自己的本地内存，通过ICN连接。各local memory在物理上分布于不同节点中，但软件层面为统一地址空间。

按是否维护cache一致性，分为两类：

- NC-NUMA（non cache）：不自动维护缓存一致性。如果不同cache中缓存统一数据，不自动保持一致。
- CC-NUMA（coherent cache）：支持缓存一致性。硬件复杂度高，但常见。

### COMA

COMA（cache only memory access）是NUMA的特殊形式。普通NUMA中，内存块通常有固定的home node；但COMA中可以根据使用情况在节点间迁移。每个数据初始可以分配在任意位置。

COMA的优点为，如果某个数据被某个节点频繁使用，可以移到该节点附近，从而降低访问延迟。但实现复杂，必须对每个节点维护directory。

## 缓存一致性问题

多核系统中，内存中同一数据在不同cache中有多份拷贝。其他处理器写后，本地cache中的数据过期，即cache coherence problem。

两种一致性问题：

- Cache coherence：针对同一地址，所有处理器看到的写入顺序应该一致
- Memory consistency：不同地址间的访问顺序。

Cache coherence protocol分为bus snooping protocol和directory based protocol两类。

### Snoopy Coherence Protocol

监听一致性协议用于UMA。所有处理器监听总线上的访问请求；当某个private cache写数据后，立刻在总线广播信息；其他处理器监听到信息后执行对应操作，如置为invalid、或同步更新本地拷贝、或将修改后数据写回。

Snoopy协议处理写操作时，有write invalidate和write update两种策略，前者更常用。

下面为几种常见的write invalidate协议。

- **MSI协议**：每个block有invalid、shared和modified三种状态，modify下其他拷贝和内存中均无效。每次读后，modify块的数据写回，所有拷贝块的状态变为shared；每次写后，当前cache块为modified、其他拷贝块为invalid。
- **MESI协议**：相比于MSI协议，增加exclusive状态，表示某个block只存在于一个cache中、且未被修改。当某个块第一次被读时，状态为exclusive；此后再被其他处理器读，则都变为shared；写操作同MSI。
- **MOESI协议**：相比于MESI协议，增加owned状态，表示当前cache中为写后的脏数据，其它cache可以有副本。modify状态下其他cache读，则当前cache变为owned，将数据发送给读的cache，其他cache从invalid变为shared。

MESI和MSI：MSI中，每次写需要在总线上广播，以将其他拷贝置为invalid；而MESI中，E状态下写不需要广播，对私有数据的读后写更高效。

MOESI和MESI：MESI中，M状态下其他cache读，必须写回内存，再全部为S；而MOESI中，允许脏数据被共享，但指定一个Owner负责维护最新值，减少写回的开销。

### Directory Protocol

目录协议用于NUMA。维护一个directory，记录每个内存块当前被哪些节点缓存、状态是什么、是否被修改。当某个private cache写后，在目录中查到哪些节点有副本，再point-to-point给这些节点发送invalid信息。

每个cache block有invalid、shared和modified三种状态，更新同MSI。

每个directory block有三种状态（和cache block的含义不同）：

- uncached：没有任何cache有这个block
- shared：一个或多个cache有只读的共享副本，内存是最新的
- modified：只有一个cache有最新副本，内存过期

更新逻辑：

- 对shared block和uncached block：读后变为shared，写后变为exclusive。
- 对modified block：目录中记录owner字段。owner自己读写，则状态不变；其他节点读，则数据写回、全部变为shared；其他节点写，仍为modified，但owner字段转移；owner自己被替换，则写回、变为uncached。
